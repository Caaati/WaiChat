## 1. 绪论

- **研究目的与意义：** 全球化协作与远程办公的普及，使得不同母语背景参与者的在线沟通成为常态 。若采用传统的外部机器翻译器粘贴流程，会产生语境削弱、操作成本上升及端到端时延叠加等损耗 。因此，将实时会话、机器翻译与领域术语约束置于同一 Web 应用闭环内，以全双工通道承载消息，并利用提示工程与术语注入提供专名一致性约束，具有明确的工程意义与演示价值 。  
- **国内外研究现状：**
  - **通信技术：** 近年来浏览器端实时应用逐步向 WebSocket 收敛，因其在握手完成后能提供全双工帧通道，非常适合即时消息的低延迟推送 。  
  - **翻译技术：** 统计与神经机器翻译对专名一致性仍需外部约束 。大语言模型兴起后使语境感知翻译成为可能，相较于重度依赖向量检索的 RAG 方案，采用子串命中筛选候选术语的“词典式提示”在对话翻译中具有更高的性价比和可解释性 。  

## 2. 相关技术与理论基础

- **WebSocket：** 该协议在 HTTP 握手之后升级连接，使客户端与服务端能在单一 TCP 连接上持续交换数据，为服务端主动推送提供了自然语义 。  
- **关系型持久化与 Redis 缓存范式：** 系统采用典型的缓存旁路（cache-aside）思路 。MySQL 提供事务化写入与约束完整性，支持历史检索 。Redis 则用于两类结构：通过列表缓存最近消息窗口以降低数据库读压力，通过字符串快照缓存术语库以支撑高频匹配 。  
- **托管式大模型 API 与多模态技术：** 托管 API（如 DashScope）将模型推理外移，系统主要调用三类形态：轻量翻译模型路径、对话式生成路径（用于润色、智能回复、摘要）以及多模态语音理解路径（将音频作为元素提交进行 ASR 转写） 。  

## 3. 系统设计

### 3.1 系统总体设计

- **设计目标：** 在单聊场景下提供稳定低延迟的消息投递，在跨语言场景提供可控翻译与生成增强，并在专名场景下提供可维护的术语约束 。  
- **逻辑架构：** 浏览器承载表现层；应用层经 SpringBoot 提供 REST 与 WebSocket 双通道能力；数据平面由 MySQL 与 Redis 构成；智能平面由独立外部的 DashScope 提供模型服务 。  
- **关键业务流程：** 登录后前端持有服务端会话态 。消息发送采用“热数据写入→异步持久化→对端在线则推送”的流水线 。翻译时服务端基于正文子串命中术语并拼入系统提示调用模型 。  

### 3.2 技术选型分析

- **前端框架：** 选择 Vue3 与 Vite，组合式 API 易于维护复杂的聊天界面状态，且 Vite 开发态效率高 。  
- **后端与通信：** 选择生态成熟的 SpringBoot 并集成 SpringSecurity 与 MyBatis-Plus 。实时通道选择 WebSocket，以满足双向转发与推送需求 。  
- **数据与智能：** 选用 MySQL 搭配 Redis 构建数据平面，模型服务选择托管 API 以降低自建集群的运维复杂度 。  

### 3.3 数据库设计

- **概念与逻辑结构：** 核心实体包含用户、聊天消息、术语及术语别名等 。消息表支持会话双向显示状态的逻辑删除与恢复；术语表包含溯源信息以表达从系统导入的个人副本关系 。  
- **物理设计优化：** 历史消息查询以会话双方标识与时间戳为复合索引，降低回表排序成本 。  
- **安全与完整性：** 密码采用 BCrypt 摘要存储作为安全底线 。应用层避免硬编码第三方密钥，消除安全隐患 。  

## 4. 系统实现

- **用户与认证模块：** 基于 SpringSecurity 表单登录，登录成功后向前端返回含用户主键的 JSON，建立与 WebSocket 路径一致的用户标识来源 。  

- **即时通信模块：**

  - 构造对称的 Redis 会话键并设置天级 TTL，实现冷会话内存自动回收 。  
  - 消息落库通过异步线程（如 `CompletableFuture`）移出推送关键路径，保证推送的高效性 。  
  - 历史记录读取优先拉取 Redis 热数据窗口，未命中则回查数据库并回填缓存 。  

- **术语库与翻译增强：** 采用子串包含配合 Top-K 截断控制提示长度，减轻大模型注意力稀释现象 。系统与个人词库快照具有 TTL 设置，在增删改时主动刷新缓存，防止脏读 。  

- **大模型功能应用：**

  **翻译与润色：** 翻译接口利用术语对封装为模型选项；润色则显式要求保留语言一致性并注入不同语气风格 。  

  **智能回复与分析：** 提取聊天上下文生成回复，摘要功能总结长文本，分析接口则通过严格 JSON 提取结构化数据（如频率、情感）供前端渲染展示 。  

- **前端交互：** 采用左右分栏布局（左侧联系人域，右侧当前会话与消息区），无缝集成了录音组件、AI 润色工具条与表情输入等功能，提升操作直觉 。  

## 5. 总结与展望

**总结：** 本系统以开源项目 WaiChat 为载体，成功打通了“WebSocket 双通道推送 + Redis 缓存旁路 + 大模型托管 API”的技术链路，并在可控复杂度下实现了通过术语约束增强跨语言翻译的一体化体验 。  

**局限与展望：** 当前原型仍有改进空间。未来可强化 WebSocket 握手鉴权以提升安全性 ；扩展群聊数据模型与广播策略 ；将媒体对象迁移至对象存储并引入预签名 URL 访问控制 ；进一步增加 API 调用的限流熔断机制，完善系统可观测性与多租户隔离功能 。
